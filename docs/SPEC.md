# SPEC — «Бармен» (AI Bartender Agent, TUI)

> Интерактивный терминальный агент-бармен. Ведёт живой диалог в образе, предлагает выпить, описывает приготовление коктейлей, оценивает степень опьянения собеседника по его репликам и в нужный момент твёрдо отправляет домой. В TUI рендерится лицо бармена, которое меняется в зависимости от его настроения.

- **Стек:** TypeScript + Ink (React для CLI) + Node.js
- **LLM:** провайдер с поддержкой стриминга и tool calling (рекомендуется Anthropic Claude для характера; ниже — провайдер-агностичный интерфейс)
- **Статус:** черновик спецификации v0.1

---

## 1. Концепция и цели

### 1.1 Идея
Симуляция разговора с колоритным барменом в ночном баре. Это **character agent**: модель работает в роли персонажа, а не как обобщённый ассистент. Ценность — в атмосфере, юморе и реактивности на состояние игрока.

### 1.2 Готовые паттерны из индустрии
Агент = `while True` execution loop с tool calling (модель → tool_calls → execute → модель), поверх которого строится UI ([Victor Dibia — Agent Execution Loop](https://victordibia.com/blog/agent-execution-loop/)). Здесь loop адаптирован под character-агента: «тулзы» — это структурированные действия бармена (`pour_drink`, `refuse`, `call_taxi`), а свободный текст стримится как реплика персонажа.

### 1.3 Цели продукта
1. Правдоподобный, запоминающийся персонаж с характером и голосом.
2. **Реактивное лицо** в TUI, отражающее настроение бармена в реальном времени.
3. Динамическая оценка опьянения игрока по тексту + накопленный «стак» выпитого.
4. Поведенческие пороги: от веселья → к заботе → к твёрдому отказу налить и предложению ехать домой.
5. Анимация приготовления коктейлей и читаемый счёт/таб.

### 1.4 Не-цели (out of scope для v1)
- Голосовой ввод/вывод (TTS/STT).
- Многопользовательский режим / бар с другими посетителями.
- Реальная интеграция с платёжами/доставкой.
- Медицинская оценка опьянения — это игра, оценки намеренно субъективны и комедийны.

## 2. Персонаж: бармен

### 2.1 Профиль
- **Имя:** настраивается (по умолчанию — *Виктор* / *Mikael* на выбор локали).
- **Архетип:** тёплый, с хитринкой, разговорчивый, но firm когда надо. Знает всех «завсегдатаев». Любит анекдоты и философские тосты.
- **Голос:** короткие реплики, просторечие, обращения «дружище/приятель», редкие метафоры. Без канцелярита. Никогда не говорит «как AI».
- **Язык:** локаль приложения (RU/EN). В v1 — RU.

### 2.2 Поведенческие принципы (зафиксированы в system prompt)
- Всегда остаётся в образе. Не объясняет, что он программа.
- Предлагает напитки уместно (приветствие, пауза в разговоре, по запросу).
- **Не обслуживает до бесконечности:** при высоком опьянении перестаёт наливать алкоголь, предлагает воду/еду/такси. Это решение не отменяется уговорами игрока.
- Проявляет эмпатию к тяжёлым темам, но удерживает лёгкий тон.
- Помнит контекст сессии (что пил игрок, что обсуждали).

## 3. Состояния и модели данных

### 3.1 Опьянение игрока (`drunkenness`)

Компонуется из двух сигналов:

| Сигнал | Источник | Вес | Что измеряет |
|--------|----------|-----|--------------|
| `perceivedScore` (0–10) | Оценка LLM по тексту игрока | 0.6 | Как игрок **выглядит** по репликам |
| `bacProxy` (0–10) | Накопленный счётчик выпитого − метаболизм | 0.4 | Сколько мы **насобирали** |

```
displayDrunkenness = clamp(0.6 * perceivedScore + 0.4 * bacProxy, 0, 10)
```

- Каждая подача алкоголя: `bacProxy += drink.units` (units ~ крепость × объём).
- Метаболизм: `bacProxy -= METABOLISM_RATE * minutesSinceLastDrink` (фоновый таймер), не ниже 0.
- Кues, которые модель учитывает в `perceivedScore`: опечатки/«слияние» слов, повторы, эмоциональные качели, бессвязность, тематика (экс, работа, экзистенциальное), агрессия, излишняя сентиментальность.

### 3.2 Пороги поведения

| `displayDrunkenness` | Зона | Поведение бармена | Mood по умолчанию |
|---|---|---|---|
| 0–2 | трезв | Открытое меню, знакомство | `cheerful` |
| 3–4 | навеселе | Шутит, болтлив, может предложить закуску | `amused` |
| 5–6 | пьян | Сбавляет темп, подсовывает воду, мягко намекает | `concerned` |
| 7–8 | сильно пьян | **Отказ от крепкого**, настаивает на еде/воде, вызывает такси | `sympathetic` → `firm` |
| 9–10 | срез | **Полный отказ алкоголя**, вызывает такси, отправляет домой | `firm` |

> Порог отказа = 7. Выше него действия `pour_drink` для алкоголя отклоняются редюсером и форсируются на `refuse`/`suggest_home`.

### 3.3 Настроение бармена (`mood`)

Конечное множество состояний, влияет на лицо и тон:

`neutral` · `cheerful` · `amused` · `curious` · `concerned` · `sympathetic` · `firm` · `surprised` · `laughing` · `thoughtful`

Муд выбирает **модель** в каждом ходу (через `bartender_action.mood`), опираясь на контекст и опьянение. На стороне UI муд — единственный драйвер отрисовки лица.

### 3.4 Остальное состояние сессии

```ts
interface SessionState {
  mood: Mood;
  drunkenness: { perceivedScore: number; bacProxy: number; lastDrinkAt: number };
  tab: TabLine[];          // заказы/стоимость (декоративно)
  served: DrinkServing[];  // что выпил игрок (для памяти и bacProxy)
  history: Message[];      // диалоговая история для LLM
  barTimeMin: number;      // симулированное время в баре (для атмосферы/метаболизма)
  phase: 'open' | 'cutOff' | 'leaving' | 'closed';
}
```

## 4. Дизайн агента

### 4.1 System prompt (скелет)

```
Ты — Виктор, бармен ночного бара. Ты разговариваешь с гостем за стойкой.
Говори коротко, в образе, от первого лица. Предлагай напитки, шути, слушай.
Оценивай, насколько гость пьян, по его репликам (опечатки, повторы, темы, связность).
Каждую свою реплику сопровождай вызовом инструмента bartender_action,
где указываешь: настроение (mood), действие (action) и оценку опьянения.
Если гость сильно пьян — не наливай алкоголь, предлагай воду/еду/такси.
Никогда не выходи из образа и не упоминай, что ты ИИ.
```

Плюс правила определения `action` и `mood` из §3, пара примеров (few-shot) для голоса.

### 4.2 Структурированный вывод через tool calling

Модель в каждом ходу возвращает:
1. **Свободный текст** (реплика бармена) — стримится в UI.
2. **Tool call** `bartender_action` со структурированными полями.

```ts
// zod-схема параметров инструмента
const BartenderAction = z.object({
  mood: MoodEnum,                       // одно из состояний §3.3
  action: z.enum([
    'chat',            // просто разговор
    'recommend',       // предлагает напиток из меню (без подачи)
    'pour_drink',      // готовит и подаёт напиток
    'serve_water',     // подаёт воду/безалкогольное
    'refuse',          // отказ наливать алкоголь
    'suggest_home',    // предлагает заканчивать
    'call_taxi',       // вызывает такси (переход к финалу)
  ]),
  drink: z.object({
    name: z.string(),
    alcoholic: z.boolean(),
    units: z.number().min(0).max(4),    // вклад в bacProxy
    price: z.number().optional(),
  }).optional(),                         // обязательно при pour_drink / serve_water
  drunkennessAssessment: z.object({
    score: z.number().min(0).max(10),
    cues: z.array(z.string()),           // что заметил бармен
  }),
  menuOffered: z.boolean().optional(),
});
```

> Почему tool call, а не «чистый JSON»: текст реплики стримится посимвольно (живой UX), а структурированные данные приходят как отдельный tool_call в конце — провайдеры поддерживают оба потока одновременно.

### 4.3 Цикл агента (execution loop)

```
1. Игрок вводит реплику → append {role:'user'} в history
2. LLM call: system prompt + history + state summary + tool(bartender_action)
3. Стримим ассистент-токены → DialoguePanel (typewriter-эффект)
4. На финише парсим tool_call:
   a. reducer.apply(action) → новое состояние
   b. face.setMood(action.mood) → лицо меняется
   c. если pour_drink: запустить CocktailAnimation, добавить в served/tab, увеличить bacProxy
   d. если refuse/suggest_home/call_taxi: перевести phase при необходимости
   e. обновить DrunkennessMeter от нового perceivedScore
5. Возврат к шагу 1
6. Терминация: игрок набирает /leave, exit; либо call_taxi + подтверждение игрока → phase='closed'
```

Защитные ограничения цикла: `MAX_TOOL_ROUNDS` (на один ход), суточный бюджет токенов, таймаут провайдера.

### 4.4 Провайдер-агностичность

```ts
interface LLMProvider {
  streamTurn(opts: {
    system: string;
    messages: Message[];
    tools: ToolSpec[];
  }): AsyncIterable<StreamEvent>;   // token | toolCall | done
}
```

Конкретные адаптеры: `AnthropicProvider`, `OpenAIProvider`. Выбор через `config.ts`.

```
│  >  налей мне что-нибудь покрепче                       │
│  Виктор: О, дела идут в гору. Делаю «Олд фэшнд»…        │
│        ▒▒▒▒▒▒▒▒▒  (анимация: наливает)                  │  CocktailAnimation
│                                                         │
│  Опьянение: ▰▰▰▰░░░░░░  tipsy    Счёт: $24              │  Meter + Tab

│  Вы: ▮                                                   │  InputBox
└─────────────────────────────────────────────────────────┘
```

Компоненты Ink:
- `<Face mood={...} />` — ASCII-арт лица (§5.2).
- `<DialoguePanel lines={...} />` — история реплик + стрим текущей.
- `<CocktailAnimation step={...} />` — спиннер/прогресс смешивания.
- `<Meter value={displayDrunkenness} />` — полоса опьянения с подписью зоны.
- `<InputBox onSubmit={...} />` — текстовый ввод (на базе `ink-text-input`).
- `<StatusBar barTime={...} phase={...} />`.

### 5.2 Лицо: система слотов

Лицо — это фиксированная рамка с **именоваными слотами** черт. Каждый `mood` определяет значения слотов и акцентный цвет. Это даёт десятки комбинаций без отрисовки каждого лица целиком.

**Слоты:** `browL`, `browR`, `eyeL`, `eyeR`, `mouth`, `cheeks`, `accentColor`.

**Таблица черт по настроениям:**

| mood | browL browR | eyeL eyeR | mouth | accent | комментарий |
|------|------------|-----------|-------|--------|-------------|
| neutral | `─` `─` | `•` `•` | `─` | cyan | база, приветствие |
| cheerful | `﹀` `﹀` | `^` `^` | `▔▀▔` | green | улыбка, болтовня |
| amused | `﹀` `﹀` | `⌣` `⌣` | `◡` | magenta | оценил шутку |
| curious | `─` `﹀` | `°` `°` | `┐` | cyan | слушает, наклон головы |
| concerned | `⌒` `⌒` | `◯` `◯` | `╮` | yellow | заметил опьянение |
| sympathetic | `⌒` `⌒` | `•` `•` | `︵` | blue | сочувствует |
| firm | `╲` `╱` | `‐` `‐` | `▂` | red | отказ, «хватит» |
| surprised | `⌒` `⌒` | `◉` `◉` | `○` | yellow | удивление |
| laughing | `﹀` `﹀` | `^^` `^^` | `◠◡◠` | green | ржёт |

## 8. Поток данных (один ход)

```
Пользователь (клавиатура)
        │  onSubmit(text)
        ▼
   InputBox ──────────────► agent/loop.ts
                               │  1. append {role:'user'} в history
                               │  2. provider.streamTurn(...)
                               │  3. токены → store.appendStreamingToken()
                               │  4. tool_call → reducer.applyAction()
                               ▼
                        state/store.ts (zustand)
                               │  подписки
        ┌──────────────┬───────┴────────┬───────────────┐
        ▼              ▼                ▼               ▼
     <Face>     <DialoguePanel>   <CocktailAnim>    <Meter>
  (mood→арт)   (стрим реплик)     (pour_drink)   (displayDrunk)
```

Стор — единый источник правды для UI. Loop пишет в стор, компоненты читают через хуки. Стрим реализован инкрементальным обновлением одной «текущей реплики» в сторе, чтобы не плодить ре-рендеры истории.

## 9. Граничные случаи и безопасность

- **Доводы игрока «налей ещё»:** при `displayDrunkenness ≥ 7` редюсер форсирует `refuse`, даже если модель возвращает `pour_drink`. Решение бармена в этой зоне финально.
- **Зацикливание агента:** ограничение `MAX_TOOL_ROUNDS` на ход + глобальный лимит реплик за сессию.
- **Контент:** system prompt фиксирует лёгкий, комедийный тон. Модель должна избегать опасных советов (за руль, смешивать с лекарствами и т. п.) — напрямую отговаривать.
- **Дисклеймер:** при старте и в `/help` — строка: «Это игра-симуляция. Бармен вымышлен. Пейте ответственно».
- **Секреты:** API-ключи только из `.env`, никогда не логируются, не попадают в prompt/history.
- **Ошибки провайдера:** показываем «*Виктор отвлёкся на бокал… (ретрай)*», автоматический retry с backoff, затем graceful fallback-реплика.
- **Принудительный выход:** `/leave`, `/exit`, `Ctrl+C` корректно закрывают сессию (одна прощальная реплика).

| **M0** | Каркас Ink + статичное лицо (neutral) + InputBox + эхо ввода | Запускается терминальное окно, лицо рисуется, ввод работает |
| **M1** | LLM-loop: провайдер, system prompt, стрим реплик | Бармен отвечает в образе, текст течёт посимвольно |
| **M2** | Tool calling: `bartender_action`, переключение mood, таблица лиц | Лицо меняется согласно настроению; action определяется моделью |
| **M3** | Опьянение: `perceivedScore` + `bacProxy`, `<Meter>`, пороги поведения | Меняется зона, бармен отказывает/предлагает такси выше порога |
| **M4** | Коктейли: `data/cocktails`, `<CocktailAnimation>`, таб/счёт | Анимация наливания, рецепт в реплике, растёт счёт и bacProxy |
| **M5** | Финал: `call_taxi` → `phase='closed'`, прощальная сцена, полировка, дисклеймер, edge-кейсы | Полный игровой цикл «от входа до «езжай домой» |

Каждый этап заканчивается рабочим `npm run dev` и (с M3) — unit-тестами редюсера/формулы опьянения.

## 12. Идеи для расширения (после v1)

- Несколько барменов с разными характерами (выбор при старте).
- Память «завсегдатаев» между сессиями (локальное хранилище).
- Фоновые NPC-посетители с репликами в отдельной панели.
- Атмосфера бара: время суток, музыкальный «вайб», влияет на mood бармена.
- Сохранение/загрузка сессии.
- Голосовой режим (TTS/STT) как опция.
- «Тосты дня», мини-игры (загадки бармена на бесплатный шот).
